# Vision & Differentiators

## What Gobbi Is

Gobbi is an installable harness for Claude Code. Users install it into their project and it orchestrates their Claude Code workflow — discussion, planning, execution, and evaluation — through a single conversational entry point.

## Benchmark: GSD

Gobbi benchmarks [GSD (get-shit-done)](https://github.com/gsd-build/get-shit-done), the most comprehensive Claude Code harness available. GSD is powerful: 57 commands, 56 workflows, 18 agents, 5 hooks, a TypeScript SDK, multi-runtime support. But GSD's power comes with learning cost.

## Three Differentiators

### 1. No Study Required

**GSD problem:** Users must learn 57 slash commands, understand the `.planning/` state system, memorize the phase workflow (discuss → research → plan → execute → verify), and know which command to use when.

**Gobbi approach:** Users just talk. One entry point. Gobbi detects intent, categorizes complexity, and routes to the right internal workflow. The user never needs to learn gobbi's vocabulary, commands, or phases. Gobbi manipulates everything based on systematic harness engineering — the system is sophisticated internally, invisible externally.

### 2. Claude Code Only

**GSD problem:** GSD supports 8+ runtimes (Claude Code, OpenCode, Gemini CLI, Codex, Copilot, Cursor, Windsurf, Antigravity). This multi-runtime abstraction adds complexity and limits what GSD can do with any single runtime.

**Gobbi approach:** Claude Code only. This allows gobbi to exploit every Claude Code feature deeply: native Agent tool, TaskCreate, memory system, hooks, settings, worktrees. No abstraction layers. No lowest-common-denominator compromises.

### 3. Safe Hacking

**GSD problem:** GSD offers config.json toggles for customization, but no isolation. Users who want deeper changes must fork the system or edit core files, risking breakage on updates.

**Gobbi approach:** Core gobbi skills live in `gobbi-*/` directories and are never touched by users. User customizations live as patch files in `gobbi-hack/`, generated by gobbi from natural conversation. Patches overlay core behavior without modifying it. Users can hack freely — gobbi's core remains intact and updatable.

## Design Principles

1. **Discuss first, never act on assumptions** — Every task starts with proportional discussion
2. **Adaptive over procedural** — Workflow shapes itself to the task, not the other way around
3. **Gotchas are the highest-value knowledge** — Mistakes recorded once are never repeated
4. **Detail is everything** — Vague prompts produce vague work; gobbi refines before executing
5. **Split narrow, execute step by step** — Small tasks, one at a time, verified before moving on
